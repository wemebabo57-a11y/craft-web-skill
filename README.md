# craft-web

A Mavis skill that makes web frontend work look like it was written by a real maintainer, not generated by default AI.

## What it does

`craft-web` gives the model a set of **constraints** (what *not* to do) for any web frontend task — building, fixing, refactoring, optimizing. It does **not** prescribe a fixed page structure or reply shape.

The constraints come in three layers:

| Layer | Covers |
|---|---|
| **Voice** | No openers, no empty apologies, no "let me think" preamble, no user-blaming on bug reports. |
| **Visual** | No purple gradient, no glassmorphism, no emoji in UI copy, restrained color/type/spacing, real type systems. |
| **Code** | Business-semantic naming, no `console.log` residue, three states for data views (loading/empty/error), real a11y, no secret leakage. |

## When it triggers

The model loads this skill when the user asks anything that touches web frontend code, styling, or UX. Examples:

- "Build me a SaaS landing page"
- "Fix this UI bug — the button does nothing"
- "Optimize the interface, it looks too AI"
- "数据库连不上"
- "帮我修个网页"
- "加点动效" / "改改样式"

## When it does NOT trigger

- Pure algorithm or system-design questions
- Backend-only scripts without UI
- CLI tools, data pipelines
- Research / writing tasks
- Desktop apps, mobile native, game dev

## File structure

```
/workspace/craft-web/                ← this README (human-facing docs)
└── README.md

/workspace/.skills/craft-web/        ← the skill package (LLM-facing)
└── SKILL.md                         ← only file; syncer uploads automatically
```

Note: the skill package deliberately contains only `SKILL.md`. No `README.md`, no `CHANGELOG.md`, no `install.sh`, no `.env`. Adding those would pollute model context and break the syncer's expectations.

## Design rationale

**Constraints, not a template.** The most common failure mode for AI-written websites is *template-matching*: see "SaaS landing" → auto-spawn header/hero/logo-bar/feature/workflow/pricing/cta/footer. `craft-web` actively forbids this. The model should read the actual request, then decide.

**Pre-ship checklist, not post-mortem audit.** A baseline LLM often writes code, then lists its own concerns afterward. `craft-web` puts the 8-item checklist (XSS, secrets, a11y, states, responsive, visual restraint, cleanup, naming) in front of the writing, so the code ships clean.

**Match verbosity to the task.** A 1-line bug fix doesn't get a 5-bullet explanation. A full landing rewrite does. The skill explicitly tells the model not to manufacture ceremony.

## How to evaluate

To verify a change is improving the skill (not regressing), run a producer-vs-baseline comparison:

1. Pick a real user prompt that should trigger the skill.
2. Run two parallel agents:
   - **Producer**: load `SKILL.md`, answer the prompt.
   - **Baseline**: answer the same prompt without loading the skill.
3. Compare on:
   - Did the skill produce a more structured/complete result?
   - Did it avoid unnecessary ceremony or template-matching?
   - Was the pre-ship checklist visible in the producer's reasoning?
   - Any regressions vs baseline (length, focus, quality)?

Stop iterating when the producer clearly beats the baseline, or when two rounds produce no meaningful gain.

## Calibration anchors

When the model needs a visual reference, it should look at:
- `linear.app`
- `vercel.com`
- `stripe.com/docs`
- `github.com`
- `rauchg.com`

Not Dribbble. Not "AI-generated landing page" inspiration boards.

## Updating the skill

1. Edit `/workspace/.skills/craft-web/SKILL.md`.
2. Keep it under 500 lines. If a section grows, move it to `references/<topic>.md` and link to it.
3. Re-run the producer-vs-baseline eval after any rule change.
4. The syncer uploads the directory to OSS at session end — no manual deploy.

## Anti-patterns to avoid when modifying

- Adding `README.md` / `CHANGELOG.md` / `install.sh` / `.env` to the skill package
- Writing ALWAYS / NEVER / MUST rules without explaining why
- Stuffing the description with keywords instead of trigger phrases
- Duplicating the trigger list inside the body
- Adding scripts for things the LLM can do directly

See `skill-creator/references/anti-patterns.md` for the full list.
